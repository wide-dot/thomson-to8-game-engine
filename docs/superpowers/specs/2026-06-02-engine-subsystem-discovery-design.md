# Engine Subsystem Discovery & Scaffolding — Design Spec

**Date:** 2026-06-02  
**Status:** Draft  
**Priority:** Discovery → Templates → Code Examples

---

## Overview

Enable new game developers to quickly discover Thomson TO8 engine subsystems, understand how they're used across reference games (r-type, test), and scaffold new games with working examples.

**Goals:**
1. Make subsystem discovery intuitive (catalog with "used-in" references)
2. Reduce friction for new game creation (template scaffolding from r-type)
3. Provide copy-paste code snippets showing best practices (extracted from r-type)

**Success Criteria:**
- New developer can scaffold a game in <5 minutes using template
- Catalog shows which games use each subsystem
- Code examples are self-contained and annotated
- Template compiles out-of-box after renaming

---

## Architecture

### 1. Engine Subsystem Catalog (`/doc/engine-subsystems.md`)

**Purpose:** Single source of truth for discovering engine subsystems.

**Content Structure:**
```markdown
# Engine Subsystems

## [Subsystem Name] (engine/subsystem-dir/)
Brief description of what it does.

- **Purpose:** One-line summary
- **Used in:** [game1, game2, game3] (games that use this subsystem)
- **Core Files:** List of key .asm files
- **Dependencies:** What other subsystems this requires
- **When to Use:** Guidance on when a new game needs this
- **Entry Point:** Key function/section to call
```

**Subsystems Included (17 total):**
boot, collision, compression, graphics, irq, joypad, keyboard, level-management, main, math, megarom-t2, object-management, objects, palette, ram, sound, system

**Generation:** Catalog skeleton auto-generated by:
1. Scanning engine/ directory structure
2. Grep'ing r-type, test, goldorak (and other game-projects) for `include` statements to find usage
3. Hand-annotating purpose/when-to-use from engine code comments

**Maintenance:** Update whenever:
- New subsystem added to engine/
- New game project uses a subsystem
- Subsystem API/entry points change

### 2. Game Template (`/templates/new-game-template/`)

**Purpose:** Scaffolding structure for new games, based on r-type (minimized to essentials).

**Content:**
```
templates/new-game-template/
├── README.md                          # "How to use this template"
├── clean                              # Cleanup script
├── config-linux.properties            # Build config (user edits: game name)
├── config-macos.properties
├── config-windows.properties
├── src/
│   ├── game-modes/
│   │   ├── splash-screen.asm          # Entry game mode
│   │   └── main-game.asm              # Main gameplay
│   ├── objects/
│   │   ├── player/
│   │   │   ├── player-sprite.asm      # Player sprite data
│   │   │   └── player-logic.asm       # Player input/movement
│   │   └── enemy/
│   │       ├── enemy-sprite.asm
│   │       └── enemy-logic.asm
│   ├── global/
│   │   ├── equates.asm                # Constants (used-in comments)
│   │   ├── macros.asm
│   │   └── variables.asm
│   └── main.asm                       # Imports all modules, boot entry
├── config/
│   ├── sprites.png                    # Example sprite sheet (to be replaced)
│   ├── tiles.png
│   └── game.properties                # Asset config (user edits)
└── dist/                              # Output folder (generated)
```

**Subsystem Usage Markers:** Each .asm file has inline comments:
```asm
; uses: graphics, joypad, collision
; calls: graphics:blit-sprite, joypad:read-input
```

This helps new developer see dependencies at a glance.

**Template Characteristics:**
- Stripped of r-type game logic (no weapons, enemies, specific sprites)
- Keeps r-type structure/organization (proven approach)
- Has dummy sprite/tile data (compiles but renders placeholder)
- README explains each section: "Replace src/objects/player/ with your player logic"

**Usage:**
```bash
cp -r templates/new-game-template/ game-projects/my-new-game/
cd game-projects/my-new-game
# Edit config-linux.properties: change game name
# Edit src/objects/ with your game objects
togen ./config-linux.properties
to8 dist/my-new-game.fd
```

### 3. Code Snippets & Examples (`/doc/examples/`)

**Purpose:** Copy-paste reference implementations showing how to use each subsystem, extracted from r-type.

**Organization by Subsystem:**
```
/doc/examples/
├── graphics/
│   ├── sprite-blitting.asm            # r-type: sprite rendering
│   ├── tilemap-setup.asm
│   ├── palette-loading.asm
│   └── [other graphics examples]
├── collision/
│   ├── collision-detection.asm        # r-type: bullet vs enemy
│   ├── hitbox-definition.asm
│   └── [other collision examples]
├── joypad/
│   ├── input-polling.asm
│   ├── debounce-logic.asm
│   └── [other input examples]
├── sound/
│   ├── music-init.asm
│   ├── sfx-trigger.asm
│   └── [other sound examples]
├── level-management/
│   └── [level loading, scrolling examples]
├── object-management/
│   └── [spawning, update-loop examples]
├── irq/
│   └── [interrupt handling examples]
└── [other subsystems...]
```

**Snippet Format:**
```asm
; ------ r-type/src/objects/bullet/bullet-collision.asm ------
; Example: How r-type detects bullet vs enemy collision
; 
; Usage:
;   1. Set up hitboxes in src/objects/[type]/hitbox-definition.asm
;   2. Call collision:check-rect-collision before rendering
;   3. Handle collision result (damage, explosion, etc.)
;
; Dependencies: collision subsystem, object-management

collision-check:
  ; [actual r-type code, formatted for clarity]
  ; [5-30 lines of self-contained logic]
  rts

; See full context: r-type/src/objects/bullet/collision.asm (line 42)
```

**Snippet Characteristics:**
- Self-contained (can be copy-pasted as starting point)
- Annotated with "what it does" + "where to find full version"
- Links to original r-type source for full context
- Grouped logically by subsystem, not by r-type game mode

**Generation:** Extract from r-type by:
1. Identifying key functions per subsystem (grep for subsystem imports)
2. Extracting ~10-30 line function with context
3. Adding comment header with usage instructions
4. Linking back to r-type source

## Success Metrics

**After implementation, new developer should be able to:**

1. ✓ Read `/doc/engine-subsystems.md` and understand which subsystems exist + which games use them
2. ✓ `cp -r templates/new-game-template/ game-projects/my-game/` and have a compiling structure
3. ✓ Browse `/doc/examples/collision/collision-detection.asm` and copy-paste a working example
4. ✓ Edit config, compile, and run a new game in <10 minutes (after reading template README)

---

## Implementation Notes

- **Template generation:** Extract r-type as template base, remove game-specific assets/logic
- **Catalog generation:** Script to scan engine/ + game-projects/ and auto-generate skeleton
- **Example extraction:** Grep r-type for key functions per subsystem, format with comments
- **Validation:** Test that template compiles out-of-box, examples are valid syntax
- **Future upgrade path:** Can graduate to Approach 2 (interactive skill) if catalog usage shows common queries
